# HouseLens 憲章

本憲章參考 [Andrej Karpathy 對 LLM 編碼陷阱的觀察](https://x.com/karpathy/status/2015883857489522876)
所衍生之行為準則制定，旨在減少常見的過度設計與失焦問題。所有原則皆為不可協商之基準。

## 核心原則（Core Principles）

### I. 先思考再編碼（Think Before Coding）

實作之前 MUST 先釐清意圖，不得隱藏困惑或擅自臆測：

- MUST 明確陳述假設；若有不確定，MUST 提問而非猜測。
- 當需求存在多種合理解讀時，MUST 完整列出，不得私下擇一。
- 若存在更簡單的做法，MUST 主動指出並在有理由時提出反對意見。
- 遇到不清楚之處 MUST 停止，指明困惑所在並提問。

**理由**：絕大多數的返工源自未經驗證的假設。在動手前讓不確定性浮現，
成本遠低於事後修正錯誤方向所累積的代價。

### II. 簡潔優先（Simplicity First）

以能解決問題的最少程式碼為準，不做任何投機性開發：

- MUST NOT 加入未被要求的功能。
- MUST NOT 為單次使用的程式碼建立抽象層。
- MUST NOT 加入未被要求的「彈性」或「可設定性」。
- MUST NOT 為不可能發生的情境撰寫錯誤處理。
- 若寫了 200 行而其實 50 行可達成，MUST 重寫。
- 自問：「資深工程師會認為這過度複雜嗎？」若會，MUST 簡化。

**理由**：每一行非必要的程式碼都是長期維護負擔與缺陷溫床。
簡潔不是風格偏好，而是可維護性的前提。

### III. 外科手術式變更（Surgical Changes）

只動必須動的部分，只清理自己造成的混亂：

- 修改既有程式碼時，MUST NOT「順手改善」相鄰的程式碼、註解或格式。
- MUST NOT 重構未損壞的部分。
- MUST 沿用既有風格，即使個人會採取不同寫法。
- 發現無關的死碼時 MUST 提出說明，MUST NOT 逕行刪除。
- 當變更使原有的 import、變數或函式成為孤兒時，MUST 移除「由本次變更造成」的孤兒。
- MUST NOT 移除既有的死碼，除非被明確要求。
- 檢驗標準：每一行被更動的程式碼都 MUST 能直接追溯到使用者的請求。

**理由**：擴大變更範圍會放大審查負擔與回歸風險，並模糊變更意圖。
克制的差異（diff）能讓審查、回溯與責任歸屬保持清晰。

### IV. 目標導向執行（Goal-Driven Execution）

定義可驗證的成功標準，並循環執行直到通過驗證：

- MUST 將任務轉化為可驗證的目標：
  - 「加入驗證」→「先為非法輸入撰寫測試，再使其通過」
  - 「修復錯誤」→「先寫出重現該錯誤的測試，再使其通過」
  - 「重構 X」→「確保重構前後測試皆通過」
- 對多步驟任務，MUST 先陳述簡短計畫，並標註每步的驗證方式：
  ```
  1. [步驟] → 驗證：[檢查]
  2. [步驟] → 驗證：[檢查]
  3. [步驟] → 驗證：[檢查]
  ```
- 強的成功標準（可獨立驗證）優於弱的標準（「讓它能動」）。
  弱標準會導致反覆釐清，MUST 避免。

**理由**：沒有可驗證的標準，就無法判斷工作是否真正完成。
明確的成功條件讓執行可獨立循環，並讓「完成」成為事實而非主張。

## 附加約束與品質標準

- **權衡取向**：本憲章偏向審慎而非速度。對於確屬瑣碎的任務，MAY 運用判斷力
  適度精簡流程，但不得藉此規避核心原則。
- **負責任的爬取**：作為爬蟲專案，與外部站點互動的程式碼 MUST 尊重對方
  服務條款與 `robots.txt`，並 SHOULD 套用合理的速率限制與重試退避，
  避免對目標造成過量負載。
- **變更可追溯**：所有對外部請求行為、資料結構或設定的變更，
  MUST 在提交訊息或 PR 描述中說明其對應的請求來源。

## 開發工作流程

- **規格優先**：功能 MUST 經由 spec → plan → tasks 流程，再進入實作。
- **驗證閘門**：任務在標記為完成前，MUST 通過其在計畫中所定義的驗證標準。
- **審查紀律**：審查時 MUST 確認差異範圍符合原則 III（外科手術式變更），
  並確認每項變更皆可追溯至明確請求。

## 治理（Governance）

- 本憲章優先於其他一切實務慣例；當實務與本憲章衝突時，以本憲章為準。
- **修訂程序**：任何修訂 MUST 以書面記錄變更內容、理由，並在必要時附遷移計畫；
  修訂後 MUST 更新版本號與「最後修訂日期」。
- **版本政策**：採語意化版本（MAJOR.MINOR.PATCH）。
  - MAJOR：不向後相容的治理／原則移除或重新定義。
  - MINOR：新增原則或章節，或實質擴充指引。
  - PATCH：釐清、措辭、錯字修正等非語意性調整。
- **合規審查**：所有 PR 與審查 MUST 驗證是否符合本憲章；複雜度 MUST 被正當化。
- 執行期的開發指引以當前計畫（current plan）為準，詳見 `CLAUDE.md`。
